# 星坠幸存者开发基线

基线编号：`SF-2026-07-11`  
基线日期：2026-07-11  
引擎：Cocos Creator 3.8.8  
目标：抖音小游戏，竖屏 720×1280

## 1. 基线作用

本基线回答四件事：当前游戏已经实现什么、代码由谁负责、开发必须遵守什么、哪些承诺尚未闭合。以后所有功能、平衡、UI 和美术开发都从本页进入，不再以聊天记录、旧截图或带日期的历史报告作为现状依据。

## 2. 权威顺序

遇到冲突时按以下顺序处理：

1. 根目录 `AGENTS.md`：不可违反的产品和工程约束。
2. 实际运行源码与 catalog：当前行为、数据和资源 key。
3. 可重复执行的测试和构建产物：自动化合同与平台结果。
4. 本基线专题文档：对源码的解释、操作流程和验收要求。
5. GDD、专项设计稿：目标体验和待实现设计。
6. 带日期的审查、计划、平衡报告：历史资料。

如果第 1 项与第 2 项冲突，不得默默选择一边。先登记到 [`BASELINE_GAPS.md`](./BASELINE_GAPS.md)，明确是修实现还是修约束，再同时更新代码、测试和文档。

## 3. 当前产品边界

- 单人竖屏自动射击生存肉鸽。
- 玩家控制移动；主武器自动索敌射击。
- 出战配置固定为 1 把主武器、4 个固定部位装备和 1 把独立副武器。
- 主武器在机库替换，不能卸到 0 把；副武器不占装备槽。
- 战斗以无尽波次推进，死亡或主动撤离后结算。
- 击杀直接获得 XP；合金、材料和宝箱通过战场拾取系统处理。
- 局内成长来自升级三选一和 6 格战场商店；局外成长来自装备、蓝图、主武器和副武器系统。

完整运行事实见 [`GAMEPLAY_MECHANICS.md`](./GAMEPLAY_MECHANICS.md)。

## 4. 当前内容规模

| 内容 | 当前规模 |
|---|---:|
| 主武器 | 17 个家族 × 10 个变体 = 170 |
| 被动装备 | 44 个蓝图 × 5 个品质 = 220 |
| `EQUIPMENT` 总实例 | 390 |
| 副武器 | 15 把，T1-T5 |
| 本局道具 | 65 个单属性蓝图 × 5 tier = 325 |
| 升级选项 | 4 类 × 3 种 = 12 |
| 基础怪 | 10 个家族 × 11 个变体 = 110 |
| 小 Boss / 大 Boss | 5 / 5 |
| Cocos 场景 | 1 个：`assets/scene/Main.scene` |

数量必须从 catalog 生成或验证，不在多个文档中手工维护第二份数据源。

## 5. 工程边界

- `RogueShooterGame.ts` 是入口和总编排器。
- `core/` 放共享类型、基础属性、资源与可测试公式。
- `catalogs/` 定义静态内容；Manager 执行运行机制。
- `state/combatState.ts` 是当前显式状态对象。
- `enemy/`、`projectile/`、`pickup/`、`shop/`、`offhand/` 分别拥有运行时实体与规则。
- UI 全部由代码创建，当前同时存在静态 `PanelManager + UIHelpers` 和动态 `UIManager + PopupBase` 两套路径。
- 场景 JSON 和 UUID 引用不得手工编辑。

详见 [`PROJECT_ARCHITECTURE.md`](./PROJECT_ARCHITECTURE.md)。

## 6. 资源边界

- `assets/resources/` 是运行时资源，进入构建和包体。
- `assets/art_source/` 是源文件、候选图和处理结果，不应被游戏直接加载。
- 同名换图保留原 `.meta`；新增资源必须生成唯一 UUID。
- UI 图标不会自动扫描，必须同时接入 loader 与业务映射。
- 玩家、怪物、武器和 VFX 都有明确尺寸、帧数、文件名和 key 合同。

详见 [`UI_SYSTEM.md`](./UI_SYSTEM.md)和 [`ART_REPLACEMENT_GUIDE.md`](./ART_REPLACEMENT_GUIDE.md)。

## 7. 未闭合能力

本基线不是“全部完成”的声明。永久材料闭环、副武器材料/数值、若干副武器机制、主武器攻击距离、过热机制、结算保存、广告日切和部分美术接线仍有明确缺口。

所有缺口、严重度和验收条件以 [`BASELINE_GAPS.md`](./BASELINE_GAPS.md) 为唯一清单。未关闭的 P0/P1 项不得在宣传、提审材料或其他文档中描述为完整能力。

## 8. 变更纪律

每个变更至少完成：

1. 找到实际运行时权威代码。
2. 保持修改范围与模块所有权一致。
3. 增加能失败于旧行为的测试。
4. 按风险运行类型检查、测试、真实 Cocos/CDP 和包体检查。
5. 更新对应基线文档；有差异时更新 gap 状态。
6. 不覆盖工作区中不属于当前任务的改动。

数值和公式修改的最低验证要求仍由根目录 `AGENTS.md` 约束。

## 9. 基线升级

只有满足以下条件才能发布新基线编号：

- 关键事实已从当前源码重新审计。
- `npm run typecheck` 与 `npm test` 通过。
- 涉及运行表现时完成真实 web-mobile/CDP 验证。
- 抖音产物通过 `npm run build:check` 与 19 MiB 守门线。
- 新增/关闭的缺口已更新。
- README、AGENTS 和专题文档链接无断链。

新基线在本文顶部更新编号与日期，旧审查材料保留为历史，不反向改写其当时结论。
