# 星坠幸存者游戏设计基线

基线日期：2026-07-11
项目：`Starfall Survivor / 星坠幸存者`
平台：抖音小游戏，手机竖屏
引擎：Cocos Creator 3.8.8

本文定义产品愿景、当前系统范围和目标体验，不承担运行时数值表职责。当前公式、内容数量、掉落、价格、波次和装备规则统一查阅 [`GAMEPLAY_MECHANICS.md`](./GAMEPLAY_MECHANICS.md)。实现与设计尚未闭合的项目统一查阅 [`BASELINE_GAPS.md`](./BASELINE_GAPS.md)。

## 1. 产品定位

《星坠幸存者》是一款单人、竖屏、自动射击的生存肉鸽游戏。玩家扮演在异星战场持续出击的幸存者，只控制移动和关键流程选择，主武器自动索敌开火，副武器自动协同。

游戏的核心吸引力来自三层反馈：

1. 战斗中快速判断走位、敌群和拾取路线。
2. 单局内通过升级和道具形成清晰的成长方向。
3. 出击后通过机库装备、武器和合成系统建立长期追求。

一句话体验：

> 在不断增密的异星怪潮中边走边打，把一次次被包围的危机转化成更强的火力组合，并决定何时带着成果撤离。

## 2. 设计支柱

### 2.1 低操作门槛，持续决策

- 玩家不需要手动瞄准或持续点击射击。
- 移动、走位、拾取路线和撤离时机是主要即时决策。
- 升级三选一、商店购买、配装和副武器选择提供构筑决策。
- 操作简化不能等于没有选择；玩家应持续感受到“我这样走、这样选，所以局势改变了”。

### 2.2 主副武器协同

- 主武器是主要输出身份，决定攻击节奏、弹道和核心机制。
- 副武器自动运行，用于补足清群、控场、爆发、续航或防御短板。
- 出战语义固定为一把主武器与至多一把副武器同时生效。
- 战斗中不进行主武器切换，避免竖屏触控负担和配装语义混乱。

### 2.3 清楚的局内成长

- 击杀直接推动经验成长，不要求追逐经验球。
- 升级提供角色属性三选一，选择后立即生效。
- 战场商店出售本局道具，强化当前构筑而不是替换机库武器。
- 每次成长都应在伤害、射速、范围、生存、机动或资源效率上产生可感知变化。

### 2.4 有压力的无尽推进

- 波次持续提高敌人密度、组合和强度。
- Boss 是阶段性检查点，不是整局唯一终点。
- 玩家可以主动撤离，也可能因死亡被迫结算。
- 无尽模式的目标不是承诺必然通关，而是推动更高波次、更多构筑尝试和更好的局外准备。

### 2.5 可辨识的战斗表现

- 每个主武器家族应有可区分的弹体、枪口、命中和音效特征。
- 怪物家族通过轮廓、移动、技能、颜色和战场标记形成识别。
- UI 优先支持扫描、比较和快速操作，不使用营销页式布局干扰高频流程。
- 正式美术替换必须保持玩法碰撞、资源 key 和移动端性能稳定。

## 3. 核心循环

```text
主菜单
  -> 机库整备
  -> 进入战斗
  -> 自动索敌射击与移动走位
  -> 击杀、获得 XP、拾取合金/材料/宝箱
  -> 升级三选一与战场商店构筑
  -> 波次和 Boss 推进
  -> 主动撤离或死亡
  -> 复活/结算/战利品
  -> 返回机库强化和重新配装
  -> 再次出击
```

循环中的职责划分：

| 层级 | 主要问题 |
|---|---|
| 即时战斗 | 我怎样移动，怎样保持输出并避免被包围？ |
| 单局构筑 | 当前武器最缺伤害、节奏、范围、控制还是生存？ |
| 撤离决策 | 继续挑战更高压力，还是保住本次成果？ |
| 局外成长 | 下一次出击要替换、升级或合成什么？ |

## 4. 玩家体验目标

### 4.1 首次进入

玩家应在短时间内理解：

- 角色会自动攻击。
- 拖动或方向输入可以移动。
- 怪物、掉落、生命和波次信息位于稳定位置。
- 机库中的主武器、副武器和四件装备具有不同职责。

首屏和首局不应要求阅读长说明才能开始。

### 4.2 战斗前段

目标感受：容易进入状态，快速形成第一次成长选择。

- 敌人数量足以展示自动射击爽感，但保留走位空间。
- 第一次升级和第一次可购买行为应让玩家理解局内成长循环。
- 初始武器必须能表现自己的机制，而不是只显示不同颜色。

### 4.3 战斗中段

目标感受：构筑方向开始显现，走位和目标优先级变重要。

- 怪物组合从单一追击扩展到速度、体型、远程、爆炸和控制威胁。
- 主武器、副武器、装备和本局属性应产生协同或短板。
- 玩家应能从画面和音效判断自己为何变强、为何受伤。

### 4.4 Boss 与无尽阶段

目标感受：压力检查、机制识别和风险决策。

- Boss 需要移动和空间管理，不能只是更厚的普通怪。
- 击败 Boss 应带来明确的战利品期待和阶段反馈。
- 进入更高波次后，失败应能被解释为构筑、走位或准备不足，而不是不可读的瞬间伤害。

### 4.5 结算与机库

目标感受：清楚知道本局发生了什么，以及下一步能做什么。

- 结算应区分死亡与主动撤离。
- 收益、损失、Boss 战果和可执行的机库操作应清楚呈现。
- 新获得的武器或装备要能直接比较并替换现有出战配置。
- 玩家不能陷入“必须保留一把主武器，但又无法替换”的配装死锁。

### 4.6 重玩驱动

重玩动机来自：

- 尝试不同主武器机制和变体。
- 为主武器寻找不同副武器协同。
- 解锁、合成和升级机库内容。
- 优化局内构筑和撤离时机。
- 挑战更高波次和不同 Boss 组合。

具体平衡目标和当前运行数据不写入本 GDD，统一由机制基线和真实 Cocos/CDP 报告维护。

## 5. 当前系统范围

本节只确认系统是否属于当前产品范围，不复述易变数字。

### 5.1 已有主流程

- 主菜单、机库、战斗、暂停、设置和信息面板。
- 自动索敌、自动射击、移动、镜头和触控摇杆。
- 无尽波次、普通怪、词缀怪、小 Boss 和大 Boss。
- 经验成长、升级三选一、宝箱和战场商店。
- 死亡、激励复活占位流程、主动撤离和结算。
- Boss 战利品、装备蓝图和机库整备流程。

### 5.2 已有配装与成长范围

- 主武器家族和多档变体。
- 帽子、护甲、鞋子、首饰四个被动装备部位。
- 独立副武器目录和 T1-T5 数据模型。
- 装备拥有、强化、蓝图、解锁和配装存档。
- 传奇主武器锻造界面和配方入口。
- 本局升级属性、本局道具和资源收益体系。

具体内容数量、ID、属性和当前接线状态见 [`GAMEPLAY_MECHANICS.md`](./GAMEPLAY_MECHANICS.md)。

### 5.3 已有表现范围

- 代码生成的竖屏 HUD、菜单、机库、商店、选择、复活和结算界面。
- 主武器攻击风格、程序化 Graphics、透明 PNG primitive 和专属音效映射。
- 玩家、怪物、武器、掉落、伤害数字和战斗反馈资源入口。
- BGM、通用反馈音效和触控后的音频解锁。

UI 现状见 [`UI_SYSTEM.md`](./UI_SYSTEM.md)，美术资源合同见 [`ART_REPLACEMENT_GUIDE.md`](./ART_REPLACEMENT_GUIDE.md)。

### 5.4 自动化与交付范围

- TypeScript 规则测试和 catalog 合同测试。
- Web Mobile 构建和真实 Cocos/CDP 跑局。
- 多主武器平衡 pipeline。
- 抖音小游戏构建、产物完整性和包体守门。

验证能力和限制见 [`TESTING_AND_BUILD.md`](./TESTING_AND_BUILD.md)。

## 6. 产品边界与非目标

当前基线不包含以下产品承诺：

- 多人联机、PVP、排行榜或社交系统。
- 手动瞄准、手动换弹或战斗中切换主武器。
- 开放世界探索、关卡剧情或复杂任务树。
- 已完成的正式云存档和账号系统。
- 已完成的正式广告 SDK、支付或内购系统。
- 已通过抖音审核、适龄评定或上线合规审查。
- 所有候选美术已经接线或所有占位素材已经替换。

未来增加这些范围前，必须先更新产品基线、架构、平台边界和验收标准。

## 7. 变现与平台定位

当前设计方向是免费游玩，并在复活、战前增益、刷新或收益增强等位置预留激励视频入口。

但 `AdManager` 仍是开发期模拟实现，广告位为空，成功结果带随机模拟。正式广告、日切、失败恢复、隐私和平台合规均不能视为完成。平台和存档现状见 [`SAVE_AND_PLATFORM.md`](./SAVE_AND_PLATFORM.md)。

任何变现点必须满足：

- 不观看广告仍能完成基础核心循环。
- 奖励明确，不用误导性按钮或虚假倒计时。
- 广告失败、关闭和重复回调不破坏存档或阶段状态。
- 实现完成后才可进入上架材料和宣传描述。

## 8. 未闭合基线

当前版本仍存在会影响产品承诺的未闭合项，完整清单、严重度、证据和关闭条件以 [`BASELINE_GAPS.md`](./BASELINE_GAPS.md) 为准。

主要类别包括：

- 永久资源、Boss 材料和长期消费闭环。
- 副武器专用材料、T1-T5 数值语义和部分机制行为。
- 运行时公式与测试镜像的差异。
- 武器描述、攻击距离和个别机制接线差异。
- 商店显示与实际消费差异。
- 结算保存、广告日切和旧存档兼容风险。
- 玩家、Boss、武器挂载和 UI 图标等美术资源合同缺口。

规则：

1. `OPEN` 或 `DECISION` 的 P0/P1 项不得在其他文档、宣传或提审材料中写成“完整实现”。
2. 关闭 gap 必须同时修改代码、测试、机制文档和 gap 状态。
3. 接受当前行为也必须标记为 `ACCEPTED` 并记录理由，不能静默删除问题。
4. 本 GDD 描述目标体验，不覆盖 gap 中记录的实现事实。

## 9. 体验验收

人工试玩应持续回答以下问题：

### 核心理解

- 玩家是否无需说明就理解移动和自动射击？
- 主武器、副武器和被动装备是否容易区分？
- 商店卖本局道具而非武器是否表达清楚？

### 战斗可读性

- 玩家能否区分主要怪物威胁和伤害来源？
- 武器攻击、命中、暴击、控制和 Boss 技能是否有清楚反馈？
- UI 是否遮挡玩家、Boss、掉落或底部移动区域？

### 成长与选择

- 升级选择是否有明确差异，而不是总有唯一答案？
- 商店购买是否能改变本局表现？
- 副武器是否真正补足当前主武器，而不是只增加不可读特效？

### 结算与重玩

- 玩家是否理解死亡和撤离的差异？
- 结算后是否知道能升级、替换或合成什么？
- 失败原因是否清楚，并能形成下一局计划？

人工记录模板位于 [`playtest/PLAYTEST_TEMPLATE.md`](./playtest/PLAYTEST_TEMPLATE.md)。数值结论必须由真实 Cocos/CDP 跑局支持，不能使用旧历史报告替代。
